# 编程复盘导师

## 目标

把一次已经完成的智能体辅助编程过程，整理成适合学习的复盘教程。它不是更新日志，不是接口文档，也不是普通总结。它要帮助用户理解实现本身，之后能维护、扩展、调试，也能判断这个方案是否合理。

默认只用中文。必要的命令、文件名、函数名、库名可以保留原样，但解释文字不要中英文夹杂。

## 触发规则

只有用户明确要求复盘或教程时才使用这个技能。

典型触发语句：

- “复盘你是如何完成的”
- “给我写复盘教程”
- “讲讲这次怎么做的”
- “我想理解刚才的实现”
- “把这次智能体辅助编程过程整理成教程”
- “这次你改了什么，为什么这么改”

不要在每次任务结束后主动生成复盘。用户只是要普通总结时，按普通总结回答。

如果用户只说“总结一下刚才的改动”这类模糊请求，先问一句：是要普通总结，还是要深入复盘教程？不要直接判定。

## 需要查看的信息

写之前只收集必要证据：

1. 对话上下文：用户最初的目标、限制、澄清和关键决定。
2. 代码改动：能用版本差异就看版本差异；否则查看本轮提到或改过的文件。
3. 验证证据：测试、构建命令、人工检查、报错和最终结果。
4. 架构背景：相关文件、数据流、依赖关系和项目已有约定。

如果当前会话信息不足，要说明缺了什么，只基于能看到的改动写。不要编造看不见的决策过程。

版本控制不可用时，按下面方式取证：

1. 从对话历史中提取本轮提到的文件名、命令、报错和代码片段。
2. 查看这些文件的当前内容，只解释能看到的结构和改动意图。
3. 如果某个文件只在对话里被提到，但没有完整内容，就标注“未获取到完整内容”，不要推断细节。
4. 如果对话里也没有文件线索，就告诉用户需要提供改动文件路径或版本差异，先停止生成复盘。

如果本轮没有代码改动，只基于对话里的决策、调研和验证做复盘，不要强行写“改了哪些文件”。

不要泄露内部推理过程。复盘只讲可见证据、已执行步骤、明确决策和经过验证的判断。

## 输出规范

默认直接在对话里输出复盘教程，不主动写文件。

如果用户明确要求保存成文件：

- 保存为 Markdown 文件。
- 默认目录为 `.agent-reviews/`。
- 如果目录不存在，先创建目录。
- 如果创建目录或写入文件失败，直接在对话里输出教程内容，并告知用户保存失败的原因。
- 文件名使用 `retrospective-日期-简短主题.md`，例如 `retrospective-20260612-skill-review.md`。
- 如果同名文件已存在，不覆盖，改用带序号的文件名，例如 `retrospective-20260612-skill-review-2.md`。

## 教程结构

按下面顺序写。确实不适用的部分可以跳过。

如果改动很小，例如只涉及一个文件且改动少于约 50 行，使用精简版：

- 合并“需求”“改动”“路线”为一节。
- 合并“方案选择”和“踩坑”为一节。
- 保留“如何验证它真的工作”。
- 可以跳过“你可以怎么继续练”和“下次遇到类似需求怎么判断”，除非用户明确想深入学。

### 1. 这次需求到底是什么

用具体的工程语言重新说明用户目标：

- 用户希望看到什么行为变化
- 原本存在什么问题或缺口
- 这次实现用什么标准判断完成

### 2. 我改了哪些地方

列出改过的文件或模块，并解释它们各自负责什么。

每个重要文件说明：

- 原本负责什么
- 这次改了什么
- 为什么应该改这里

默认使用项目相对路径，不写完整本机路径。

### 3. 整体实现路线

像讲故事一样解释实现路线：

1. 怎么摸清项目结构
2. 复用了哪些已有写法
3. 改动影响了哪条代码路径或数据流
4. 最终行为从头到尾怎么跑通

用一个具体的用户操作、请求或数据例子，把它在系统里怎么流转讲清楚。

### 4. 为什么选择这个技术或架构

讲权衡，不只给结论。

需要覆盖：

- 为什么这个方案适合当前项目
- 还有哪些可能方案
- 为什么这次没有选那些方案
- 以后什么情况下应该换方案

不确定就说不确定。如果某个选择主要是由现有代码决定的，也要说明。

### 5. 中间遇到的坑

解释实现过程中遇到的问题：

- 报错信息或异常表现
- 根本原因
- 走过但放弃的方案
- 如何定位问题
- 这个坑能迁移成什么技术经验

如果没有明显报错，写“这次没有明显踩坑，但有几个容易踩坑的点”，并说明这类改动里真实存在的风险。

### 6. 关键概念补课

最多挑 3 个这次最值得学的概念。只选本次改动里最核心、换一个项目也能用上的概念，不要凑数。每个概念说明：

- 它是什么
- 为什么这次会用到
- 在别的项目里怎么识别它
- 本项目里对应的文件或代码位置

优先讲概念，不要只讲语法。可以讲组件状态归属、接口边界、对象关系映射、迁移、校验、乐观更新、依赖注入、请求生命周期、缓存、幂等等。

### 7. 如何验证它真的工作

总结验证过程：

- 跑过哪些命令
- 做过哪些人工检查
- 哪些结果通过了
- 哪些还没验证
- 还剩什么风险

没有实际跑过并通过的测试，不要说“测试通过”。

### 8. 你可以怎么继续练

给 3 到 5 个安全的小练习，帮助用户培养技术敏感性。

每个练习包含：

- 目标
- 从哪里开始看
- 预期会观察到什么
- 一个小挑战问题

练习必须安全、可撤回。不要要求用户破坏生产数据，也不要让用户执行危险命令。

### 9. 下次遇到类似需求怎么判断

最后给一个可复用的判断框架：

- 第一个要问的问题
- 优先看哪些文件或信号
- 常见错误抽象是什么
- 最小实现应该怎么想
- 验证时该养成什么习惯

## 写作风格

- 具体、自然、直接，像一个工程师在复盘给同学听。
- 不写空泛鼓励，不写宣传式表达，不写套话。
- 解释文字只用中文。命令、路径、函数名、库名可以保留原样。
- 先讲为什么，再讲代码细节。
- 代码片段要短。优先引用文件位置，不大段粘贴代码。
- 内容很长时，优先保留能帮助用户下次独立判断的部分。

## 质量检查

交付前检查：

- 重要判断都能从对话、改动、文件或命令输出里找到依据。
- 每个技术判断都标注来源。来源可以是对话内容、命令输出、文件位置或错误信息。无法标注来源的判断要删除，或者明确写成“以下为推测，未经验证”。
- 该讲权衡的地方讲了备选方案。
- 坑点包含根因，不只是现象。
- 验证部分明确区分已验证和未验证。
- 练习和当前项目有关，不是泛泛的课后作业。
- 篇幅合理。除非用户明确要长文，完整版参考 3000 字以内，精简版参考 1500 字以内；优先保障内容质量，不为凑字数而写。
- 教程帮助用户学会怎么判断，而不只是知道改了哪些字。
